⚙️ Workflow Tables – Schema & Purpose

Complete table design for workflow module. Base tables + dynamic per-workflow tables.

📌 Quick Summary
• Base tables (Workflows, Steps, Lookup, Approvals, SLAs) = definitions and cross-workflow index. Created once.
• Per-workflow tables (Instances, StepInstances, Comments, Attachments...) = runtime data. Created when you publish a workflow.
• WorkflowInstanceLookup = central index. Tells us "which workflow is this item in?" for Inbox / Sent / Completed.

Step-by-Step: What Happens When

Each action creates or affects specific tables. Follow the flow below.

1 POST /api/workflows/setup-schema (run once per tenant, or during signup)
Creates: workflow.Workflows, workflow.WorkflowSteps, workflow.WorkflowInstanceLookup, workflow.WorkflowApprovals, workflow.WorkflowSlas
2 Create workflow (POST /api/workflows – add draft)
Affects: INSERT into Workflows, INSERT into WorkflowSteps
3 Publish workflow (POST /api/workflows/{id}/publish)
Creates: WorkflowInstances_{suffix}, WorkflowStepInstances_{suffix}, WorkflowInstanceSlas_{suffix}, WorkflowInstanceUserState_{suffix}, WorkflowComments_{suffix}, WorkflowAttachments_{suffix}, WorkflowForms_{suffix}, WorkflowTasks_{suffix}, WorkflowSignatures_{suffix}, WorkflowDocuments_{suffix}, WorkflowEmails_{suffix}, WorkflowAiValidations_{suffix}, WorkflowPdfAnnotations_{suffix}
Affects: UPDATE Workflows (Status = Published)
4 Start instance (POST /api/workflows/{id}/start)
Affects: INSERT into WorkflowInstanceLookup, INSERT into WorkflowInstances_{suffix}, INSERT into WorkflowStepInstances_{suffix}
5 Approve / Reject (POST .../approve or .../reject)
Affects: UPDATE WorkflowStepInstances_{suffix}, INSERT into WorkflowApprovals, UPDATE WorkflowInstances_{suffix}
6 Add comment, upload attachment
Affects: INSERT into WorkflowComments_{suffix}, WorkflowAttachments_{suffix}, etc.

1. Overview

Workflow schema has two phases:

  • setup-schema – Base tables (definitions, lookup). Created once.
  • publish – Each workflow gets its own set of tables when you publish it.

1a. Why Separate Tables?

In plain English: Think of it like a filing system. You don't mix recipe cards with invoices. Each type of data goes in its own drawer. Same here: definitions, running instances, and approvals each have their own table.
TableWhy separate? (Simple)
WorkflowsStores the blueprint (template). Rarely changes. Separate so template edits don't touch millions of running items.
WorkflowStepsSteps in the workflow (Step 1 → Step 2 → Step 3). One workflow = many steps. Separate because it's different data: "what to do" vs "who did it".
WorkflowInstanceLookupKey table. Like an index: "Which workflow is this item in?" Needed for Inbox / Sent / Completed across all workflows. Without it we'd have to check every workflow table one by one.
WorkflowApprovalsWho approved, when, and comments. Separate for easy reporting and audit.
WorkflowSlasRules (e.g. "respond in 24 hrs"). Separate from runtime tracking (actual deadlines).

1b. Why Per-Workflow Tables (Not One Big Table)?

In plain English: Imagine one huge table with all workflow types (Invoice Approval, Leave Request, Purchase Order...). Millions of rows, slow searches. Instead, each workflow type gets its own table. Like separate filing cabinets: Invoice Approvals in one, Leave Requests in another. Faster and easier to manage.
ReasonSimple explanation
SpeedSmaller tables = faster searches. No need to filter by "which workflow?" every time.
No blockingHeavy traffic on Invoice Approval doesn't slow down Leave Request. Each workflow is isolated.
FlexibilityInvoice workflow can have different columns than Leave workflow. Each table can have its own structure.
Easy maintenanceArchive or move one workflow's data without touching others.
WorkflowInstanceLookup ties them together. When you open Inbox, we check this table first to see which workflow each item belongs to, then fetch details from the right per-workflow table.

1c. Why WorkflowInstanceLookup?

In plain English: Data is split across many tables (one per workflow). Inbox needs "all items assigned to me" from ALL workflows. Lookup is the central index: it tells us InstanceId → WorkflowId so we know where to look. Without it, we'd have to query every workflow table and merge results = slow.

What it stores: InstanceId, WorkflowId, Status, AssignedToUserId, StartedBy. Small rows. Heavy data (forms, attachments) stays in the per-workflow tables.

3. Dynamic Per-Workflow Tables (created on publish)

When you publish a workflow, we create tables like WorkflowInstances_8fc95f64 (suffix = first 8 chars of workflow Id).

Why separate tables for each? Same idea as 1b: one workflow = one set of tables. Comments, attachments, forms, etc. are separate so listing instances stays fast. We load details (comments, files) only when the user opens an item.
TableWhat it holdsWhy separate?
WorkflowInstances_{suffix}Each running workflow (status, who started it)Core data. Kept lean.
WorkflowStepInstances_{suffix}Current step per instance → Inbox countOne instance = many steps. Inbox filters by AssignedToUserId here.
WorkflowInstanceSlas_{suffix}Deadline trackingUpdated often. Separate from main instance.
WorkflowInstanceUserState_{suffix}Read/unread per userOne instance = many user states. Don't bloat instance row.
WorkflowComments_{suffix}CommentsOne instance = many comments. Load on demand.
WorkflowAttachments_{suffix}Uploaded filesOne instance = many files. Load when user opens item.
WorkflowForms_{suffix}Form data (JSON)Can be large. Don't slow instance listing.
WorkflowTasks_{suffix}Sub-tasksOne instance = many tasks. Separate for clarity.
WorkflowSignatures_{suffix}Digital signaturesAudit. One instance = many signatures.
WorkflowDocuments_{suffix}Links to DMS documentsChecklist. Separate so instance stays small.
WorkflowEmails_{suffix}Emails sentLog. Don't load with instance.
WorkflowAiValidations_{suffix}AI check resultsCan be large. Load on demand.
WorkflowPdfAnnotations_{suffix}Annotations on PDFsMany per doc. Load when viewing PDF.

4. Inbox / Sent / Completed Count Logic

API: GET /api/workflows/counts

In plain English:
• Inbox = Items assigned to me, waiting for my action. We count running steps where AssignedToUserId = me.
• Sent = Items I started, still in progress. We count instances where StartedBy = me.
• Completed = All finished items (anyone). We count instances where Status = Completed.
Tables used:
Inbox → WorkflowInstanceLookup + WorkflowStepInstances_{suffix} (AssignedToUserId, Status=Running)
Sent → WorkflowInstanceLookup (StartedBy, Status=Running/Pending)
Completed → WorkflowInstanceLookup (Status=Completed)

5. Query Patterns for Inbox

-- Get inbox items (assigned to me, running steps)
-- 1. Get all workflow suffixes for tenant
-- 2. For each WorkflowStepInstances_{suffix}:
SELECT wi.Id, wi.ReferenceNumber, wi.CustomerName, wi.Status, ssi.Id AS StepInstanceId
FROM workflow.WorkflowInstances_{suffix} wi
JOIN workflow.WorkflowStepInstances_{suffix} ssi ON ssi.WorkflowInstanceId = wi.Id
WHERE ssi.AssignedToUserId = @CurrentUserId AND ssi.Status = 1  -- Running
  AND wi.Status = 1  -- Running
ORDER BY wi.LastActivityAtUtc DESC
OFFSET @Skip ROWS FETCH NEXT @PageSize ROWS ONLY;
