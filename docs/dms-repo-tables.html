📁 DMS Repository Tables – Folder Structure Metadata Design

Physical columns, FolderHierarchy, query patterns, indexing, API shape, and configurable structure.

📌 Quick Summary
• Repository = storage settings. RepositoryFolderConfig = folder structure rules (Year→Type→Vendor).
• ezca_{code}_items = actual documents. One table per tenant/entity for speed and isolation.
• Folder path = Year / InvoiceType / VendorName. Stored as separate columns for fast filtering.

Step-by-Step: What Happens When

Each action creates or affects specific tables. Follow the flow below.

1 POST /api/Dms/setup-schema (run once per tenant, or during signup)
Creates: dms.Repository, dms.RepositoryFolderConfig, dms.DocumentWorkflowLink, dms.sample_items
2 Create repository (insert into Repository with Code, Name, ItemsTableName)
Affects: INSERT into dms.Repository
3 Add folder config (optional – if repo needs custom folder levels)
Affects: INSERT into dms.RepositoryFolderConfig (Year, InvoiceType, VendorName, etc.)
4 Create items table for new repo (run CreateDmsRepositoryItemsTable.sql with suffix e.g. ezca_156)
Creates: dms.ezca_156_items (or dms.{suffix}_items)
Affects: Set Repository.ItemsTableName = "ezca_156_items" when creating repo
5 Upload document
Affects: INSERT into dms.sample_items (or dms.ezca_156_items)
6 Link document to workflow
Affects: INSERT into dms.DocumentWorkflowLink, UPDATE items table (WorkflowInstanceId)
7 Get folder tree / documents (GET folders/children, GET documents)
Affects: READ from sample_items or ezca_156_items (GROUP BY for tree)

1a. Why Separate Tables in DMS?

In plain English: Like folders in a filing cabinet. Repository = the cabinet. RepositoryFolderConfig = how folders are organized. Document items = the actual files. Each has its own table because they do different jobs.
TableWhat it does (simple)
dms.RepositoryRepository settings (name, storage path). One row per repo. Rarely changes.
dms.RepositoryFolderConfigFolder structure rules (Year→Type→Vendor, or custom). Different repos can have different structures.
dms.DocumentWorkflowLinkLinks documents to workflow instances. "This invoice is in workflow X". Kept separate so doc and workflow stay independent.
ezca_{code}_itemsActual documents. Each tenant/entity gets its own table (e.g. ezca_156_items). See 1b.
dms.FolderHierarchyOptional. Pre-computed folder tree + counts. Speeds up tree UI. See 1c.

1b. Why Per-Repository Document Tables (Not One Big Table)?

In plain English: Like having separate file drawers for each department. Instead of one huge table with all tenants' documents, we use ezca_156_items, ezca_157_items, etc. Each tenant/repo gets its own table. Faster searches, no mixing of data, easier to scale.
ReasonSimple explanation
IsolationOne tenant's heavy usage doesn't slow others. Each has its own table.
Different columnsInvoice repo may need InvoiceNumber; HR repo may need EmployeeId. Each table can have its own columns.
SpeedSmaller tables = faster folder drills. No need to filter by RepositoryId on every query.
Easy maintenanceArchive or move one repo's data without touching others.

1c. Why FolderHierarchy (Optional)?

In plain English: Two options for the folder tree:
• Real-time: Count documents on each request. Always correct, but slower when you have lots of docs.
• FolderHierarchy: Pre-calculate folder counts. Faster tree load, but needs a sync step when documents change.

FolderHierarchy is a separate table so tree queries don't scan the big document table. Document table = many writes. Folder tree = many reads. Split = faster.

2. Mandatory Fields for Folder Path

In plain English: Folders are: Year → InvoiceType → VendorName. Example: 2025 / Purchase / Acme Corp / INV-001.pdf. We store Year, InvoiceType, VendorName as separate columns so we can filter and index them quickly.

Path pattern: Year / InvoiceType / VendorName / invoicenumber.pdf

ColumnTypePurpose
YearSMALLINT NOT NULLe.g. 2025 – Folder level 0
InvoiceTypeNVARCHAR(64) NOT NULLPurchase, Sales, Credit Note – Folder level 1
VendorNameNVARCHAR(256) NOT NULLe.g. Acme Corp – Folder level 2
FileNameNVARCHAR(512) NOT NULLe.g. INV-001.pdf (stored filename)
FolderPathComputed PERSISTEDYear + '/' + InvoiceType + '/' + VendorName for display
 
FolderPath AS (CAST(Year AS NVARCHAR(4)) + N'/' + InvoiceType + N'/' + VendorName) PERSISTED

Use FolderPath for display; use Year, InvoiceType, VendorName for filtering.

Why separate columns (Year, InvoiceType, VendorName)? Filters like "2025" and "Purchase" are fast with indexes. One long path string (e.g. "2025/Purchase/Acme Corp") would need slower text matching. Separate columns = faster queries.

3. Folder Hierarchy Table (for Tree + Counts)

Pre-aggregate folder nodes and counts for fast tree rendering. Optional materialized approach.

ColumnTypePurpose
IdBIGINT IDENTITYPrimary key
RepositoryIdUNIQUEIDENTIFIERRepository
TenantIdUNIQUEIDENTIFIERTenant
YearSMALLINT NULLNULL = root/aggregate
InvoiceTypeNVARCHAR(64) NULLLevel 1
VendorNameNVARCHAR(256) NULLLevel 2
PathNVARCHAR(512)Display path: "2025", "2025/Purchase", "2025/Purchase/Acme Corp"
LevelTINYINT0=Year, 1=InvoiceType, 2=VendorName, 3=Leaf
DocumentCountINT DEFAULT 0Pre-computed count
ParentPathNVARCHAR(512) NULLParent for tree structure
 
CREATE TABLE dms.FolderHierarchy (
    Id BIGINT IDENTITY(1,1) NOT NULL,
    RepositoryId UNIQUEIDENTIFIER NOT NULL,
    TenantId UNIQUEIDENTIFIER NOT NULL,
    Year SMALLINT NULL,
    InvoiceType NVARCHAR(64) NULL,
    VendorName NVARCHAR(256) NULL,
    Path NVARCHAR(512) NOT NULL,
    Level TINYINT NOT NULL,
    DocumentCount INT NOT NULL DEFAULT 0,
    ParentPath NVARCHAR(512) NULL,
    UpdatedAt DATETIME2(3) NOT NULL DEFAULT SYSUTCDATETIME(),
    PRIMARY KEY (Id),
    UNIQUE (RepositoryId, Path),
    INDEX IX_FolderHierarchy_Repo_Parent (RepositoryId, ParentPath),
    INDEX IX_FolderHierarchy_Repo_Level (RepositoryId, Level)
);

4. Query Patterns for File Explorer UI

A. Root level (Years with counts)

SELECT Year AS Name, 
       SUM(DocumentCount) AS DocumentCount,
       CAST(Year AS NVARCHAR(4)) AS Path
FROM dms.ezca_156_items
WHERE IsDeleted = 0 AND RepositoryId = @RepoId
GROUP BY Year
ORDER BY Year DESC;

B. Children of Year (e.g. 2025)

-- Level 1: Invoice Types under 2025
SELECT InvoiceType AS Name,
       COUNT(*) AS DocumentCount,
       '2025/' + InvoiceType AS Path
FROM dms.ezca_156_items
WHERE IsDeleted = 0 AND RepositoryId = @RepoId AND Year = 2025
GROUP BY InvoiceType
ORDER BY InvoiceType;

C. Children of Year/InvoiceType (e.g. 2025/Purchase)

-- Level 2: Vendors under 2025/Purchase
SELECT VendorName AS Name,
       COUNT(*) AS DocumentCount,
       '2025/Purchase/' + VendorName AS Path
FROM dms.ezca_156_items
WHERE IsDeleted = 0 
  AND RepositoryId = @RepoId 
  AND Year = 2025 AND InvoiceType = 'Purchase'
GROUP BY VendorName
ORDER BY VendorName;

D. Documents in selected folder

-- User selected: 2025 / Purchase / Acme Corp
SELECT Id, FileName, Status, CreatedAt, WorkflowInstanceId, ...
FROM dms.ezca_156_items
WHERE IsDeleted = 0 
  AND RepositoryId = @RepoId
  AND Year = 2025 AND InvoiceType = 'Purchase' AND VendorName = 'Acme Corp'
ORDER BY CreatedAt DESC
OFFSET @Skip ROWS FETCH NEXT @PageSize ROWS ONLY;

5. Indexing for Folder Queries

-- Composite index for folder drill-down and document listing
CREATE NONCLUSTERED INDEX IX_ezca_156_items_FolderStructure 
ON dms.ezca_156_items (RepositoryId, IsDeleted, Year, InvoiceType, VendorName)
INCLUDE (Id, FileName, Status, CreatedAt, WorkflowInstanceId);

-- Filtered index for active documents
CREATE NONCLUSTERED INDEX IX_ezca_156_items_Folder_Active 
ON dms.ezca_156_items (RepositoryId, Year, InvoiceType, VendorName)
WHERE IsDeleted = 0;

6. Folder Hierarchy Maintenance

Simple choice:
• Real-time: Count on each request. Always correct. Slower with lots of docs.
• FolderHierarchy: Pre-calculate counts. Fast tree. Need to update when docs change (trigger or job).
OptionProsCons
A. Real-timeAlways correct, no syncSlower with many docs
B. FolderHierarchyFast treeMust keep in sync when docs change

8. Path Parsing and Validation

How paths map to queries: Empty = root (show years). "2025" = show types under 2025. "2025/Purchase" = show vendors. "2025/Purchase/Acme Corp" = show documents.
PathMeaning
""Root (list Years)
2025Year=2025 (list InvoiceTypes)
2025/PurchaseYear=2025, InvoiceType='Purchase' (list Vendors)
2025/Purchase/Acme CorpFull path → list documents
 
// Parse path on server
var parts = path.Split('/');
int? year = parts.Length > 0 && int.TryParse(parts[0], out var y) ? y : null;
string invoiceType = parts.Length > 1 ? parts[1] : null;
string vendorName = parts.Length > 2 ? parts[2] : null;

10. Configurable Folder Structure

If folder levels vary by repository, use dms.RepositoryFolderConfig:

ColumnTypePurpose
RepositoryIdUNIQUEIDENTIFIERRepository
LevelOrderTINYINT0, 1, 2, 3
FieldNameNVARCHAR(64)Year, InvoiceType, VendorName
DisplayNameNVARCHAR(128)UI label
 
CREATE TABLE dms.RepositoryFolderConfig (
    RepositoryId UNIQUEIDENTIFIER NOT NULL,
    LevelOrder TINYINT NOT NULL,
    FieldName NVARCHAR(64) NOT NULL,
    DisplayName NVARCHAR(128) NOT NULL,
    PRIMARY KEY (RepositoryId, LevelOrder)
);

Build dynamic queries from this config. For a fixed structure, physical columns are simpler and faster.
